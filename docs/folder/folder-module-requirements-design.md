## 1. 文档目标与适用范围

### 1.1 文档目标
- **设计**：本文件在充分依托现有 `bid-doc-system` 单体 Spring Boot 工程（已实现 `auth` 与 `common` 模块）的前提下，输出一份面向企业级文档管理系统的 **folder 模块详细需求与设计文档**。  
  目标是：在不虚构现有实现的前提下，给出 folder 模块的业务建模、权限模型、规则设计、数据库与缓存建议，并细化到后续可直接继续编写接口文档与技术实现方案。

### 1.2 适用范围
- **设计**：本文件适用于以下角色在当前项目中的工作：
  - 产品经理：评审 folder 模块的业务边界、功能范围与规则。
  - Java 后端工程师 / 架构师：据此完成接口设计、数据库设计与实现工作。
  - 前端工程师：据此理解 folder 领域模型与接口预期行为，设计前端交互与权限控制。
  - 测试工程师：据此编写测试用例与验收标准。

### 1.3 不在本次范围内的内容（设计）
- 文件本身的元数据与内容存储（属于未来 `file` 模块范畴）。
- 标签体系与标签批量导入（属于未来 `tag` 模块范畴）。
- 审批流程落地实现（例如文件夹创建/授权需要审批，仅作为后续可扩展方向）。
- 在线创建/编辑文件、第三方导入（如从云盘、OA 等导入）。
- 自动规则生成文件夹、关注/订阅通知机制的复杂策略。
- 回收站/软删除恢复机制（本期删除为直接删除）。

### 1.4 与其他模块边界（设计）
- **与 `auth` 模块**：
  - `auth` 负责用户、角色、部门、登录与会话管理，以及基于 Sa-Token 的角色鉴权能力。
  - folder 模块在鉴权、数据范围与审计上 **复用 `auth` 与 `common` 现有能力**，不重复实现用户体系。
- **与未来 `file` 模块**：
  - file 负责“文件”这一业务对象（如招标文件、合同文档）的元数据与内容。
  - folder 负责“文件夹”树与文件归属关系，是 file 的组织与权限边界。一个文件必须强绑定一个主归属文件夹。
- **与未来 `tag` 模块**：
  - tag 负责标签维度的分类与搜索。
  - folder 只负责树状层级，不与标签逻辑耦合。
- **与未来 `log` 模块**：
  - log 负责业务操作日志表及查询能力。
  - folder 在本期中 **设计自身的操作日志对象模型与需求**，落地时可与 log 模块实体/表统一或对接。
- **与未来 `search` 模块**：
  - search 负责跨对象（文件、文件夹、标签等）的全文检索与条件搜索。
  - folder 本期仅定义自身的搜索/筛选需求和基础结构，不指定底层搜索引擎实现。
- **与未来 `upload` 模块**：
  - upload 负责文件上传、分片、断点续传等能力。
  - folder 需要为 upload 提供“目标文件夹”与权限检查入口。

### 2.3 已有业务逻辑与可复用能力概览（现状）
- `auth` 模块：
  - 已实现用户注册、登录、获取当前用户、登出。
  - 已实现用户启用/禁用、部门管理（创建与列表）、角色分配/撤销与按角色查询用户。
  - 使用 `SysUser`、`SysRole`、`SysUserRole`、`SysDepartment` 等实体，采用 `IdType.ASSIGN_ID` 主键和逻辑删除字段。
  - 使用 `RoleCodeEnum` 定义统一角色编码，包含 `SUPER_ADMIN`、`FOLDER_ADMIN`、`DEPT_MANAGER`、`EMPLOYEE` 等。
- `common` 模块：
  - 统一响应体 `ApiResponse<T>`，包含 `code` / `message` / `data` / `timestamp`。
  - 统一异常体系：`BusinessException` + `ErrorCode` + `GlobalExceptionHandler`。
  - 统一登录校验与用户上下文：
    - `TokenInterceptor` 通过 `StpUtil.checkLogin()` 校验登录，从 Sa-Token Session 取 `SysUser` 与 `roleCodes`，写入 `UserContext` 和 MDC。
    - `StpInterfaceImpl` 实现 `StpInterface`，从 Session / 数据库提供角色列表。
  - 日志与 traceId：`LoggingInterceptor` 生成 `traceId`，输出请求耗时与用户信息。
  - 审计字段自动填充：`MyMetaObjectHandler` 在插入/更新时自动填充 `createdAt`/`updatedAt`/`createdBy`/`updatedBy`。

### 2.4 Sa-Token 接入方式与权限控制实现盘点（现状）
- **Sa-Token 集成**：
  - 配置于 `application.yml`，指定 `token-name: Authorization`、`token-style: uuid`、JWT 密钥、超时时间等。
  - 登录逻辑中调用 `StpUtil.login(userId)`，并在 Session 中缓存 `user` 与 `roleCodes`。
  - 注册 `SaInterceptor`，从而支持注解式权限（`@SaCheckRole` 等）。
- **角色/按钮级权限实现**：
  - Controller 层：
    - `RoleController` 上使用 `@SaCheckRole("SUPER_ADMIN")` 控制角色分配/撤销等敏感操作。
  - 拦截器层：
    - `AuthInterceptor` 根据 URL 前缀限制访问（如 `/api/v1/users/**` 仅 `SUPER_ADMIN` 可访问）。
  - **尚未发现** 基于权限编码（`@SaCheckPermission`）的细粒度按钮权限实现。
- **数据范围控制（现状）**：
  - 已有数据与上下文预留：
    - `UserContext.UserInfo` 中包含 `deptId`，并提供 `isDeptManager()`、`isFolderAdmin()` 等方法。
    - 实体中普遍存在部门相关字段。
  - **未发现** 实际的数据范围实现（无数据范围注解 / AOP 切面 / Mapper 统一封装），仅处于能力预留状态。

## 3. 模块定位

### 3.1 folder 模块在系统中的角色（设计）
- **设计**：folder 模块是“企业级文档管理系统”的 **文件组织与权限边界核心模块**，负责：
  - 维护全局唯一的文件夹树（只有一个根节点）。
  - 为所有文件对象提供主归属文件夹。
  - 在文件夹层级上实现权限控制、授权与继承策略。
  - 提供收藏、导出清单、操作日志查询等围绕“文件夹视角”的辅助能力。

### 3.2 核心职责（设计）
- 管理文件夹树结构（创建/编辑/重命名/删除/移动/复制）。
- 管理文件与文件夹的主归属关联（设计上保证：每个文件强绑定一个主归属文件夹，本期不考虑多归属）。
- 提供文件夹级授权与管理员配置能力，结合 `auth` 的用户/角色/部门体系进行权限管控。
- 提供按文件夹视角的文件清单导出与搜索/筛选能力。
- 记录并提供文件夹级操作日志，配合现有日志拦截进行审计。
- 提供用户维度的文件夹收藏能力。

### 3.3 与其他模块关系（设计）
- **与 `auth` 模块**：
  - 使用 `SysUser`、`SysRole`、`SysDepartment` 作为授权对象。
  - 通过 `UserContext` 获取当前用户身份、角色、部门，用于权限判定与数据范围控制。
- **与未来 `file` 模块**：
  - `file` 将引用 folder 的主键作为其归属目录外键。
  - 查询文件清单和导出能力以“文件夹 + 条件筛选”为主入口。
- **与未来 `tag` 模块**：
  - `tag` 可以以 folder 为范围过滤标签的应用对象，但不改变 folder 层级结构。
- **与未来 `log` 模块**：
  - folder 操作日志可映射/汇总到 log 模块，实现跨模块的统一审计查询。
- **与未来 `search` 模块**：
  - search 可以基于 folder 的树路径、名称和授权信息对文件与文件夹进行搜索。
- **与未来 `upload` 模块**：
  - upload 需要在上传目标中指定 folder，并调用 folder 的权限校验逻辑。


## 4. 业务目标与设计原则

### 4.1 建设目标（设计）
- 提供覆盖日常文档管理场景的文件夹管理能力，支撑约 3000 用户、峰值并发不超过 1200 的中小型项目。
- 在不破坏现有 auth 与 common 架构的前提下，为后续 file、tag、log、search、upload 等模块提供稳固的“目录与权限”基础。
- 兼顾当前需求与未来扩展（多租户、多项目、多根节点等）但不过度设计。

### 4.2 最小改动原则（设计）
- 不修改既有 `auth` 与 `common` 代码的行为，仅通过新增 folder 相关代码进行集成。
- 复用现有：
  - 统一响应、异常体系和日志拦截。
  - Sa-Token 配置与会话管理。
  - 审计字段与逻辑删除配置。
- 不在本期引入新的重型中间件（如消息队列、搜索引擎集群），增强项仅作建议。

### 4.3 向后兼容原则（设计）
- 所有新增接口均挂载在新的 URL 前缀（如 `/api/v1/folders/**`），不破坏现有接口。
- 权限控制在角色/权限点设计时，保持对已有角色体系（`RoleCodeEnum`）的兼容，不改变现有角色语义。

### 4.4 可扩展原则（设计）
- 通过合理的数据模型保证：
  - 后续可以添加更多授权维度（如项目、租户）而无需大规模重构。
  - 可以增加更多 folder 元数据（通过扩展字段或 JSONB），不影响核心结构。

### 4.5 安全优先原则（设计）
- 默认采用“最小权限”原则：无显式授权则只能访问自己有权访问的文件夹及其子树。
- 对所有修改类操作，统一要求登录、权限校验与审计记录。

### 4.6 审计优先原则（设计）
- 结合现有 traceId 与用户上下文，在关键 folder 操作（创建、移动、复制、删除、授权等）中记录业务级操作日志。
- 操作日志要求至少能回答以下问题：
  - 谁（用户/部门/角色）在什么时候对哪个文件夹做了什么操作、前后变化是什么。


## 5. 核心业务对象建模

> 本节为“设计”，结合现有实体与审计模式，定义 folder 相关核心对象。所有实体均遵循：`IdType.ASSIGN_ID` 主键、`createdAt/By`、`updatedAt/By`、`deleted` 审计与逻辑删除字段。

### 5.1 Folder（文件夹）
- **含义（设计）**  
  - 表示文档系统中用于组织文件的层级节点，构成唯一的全局文件夹树。

- **核心字段（设计）**（示意，非 DDL）  
  - `id`：主键，雪花 ID，非空。  
  - `parentId`：父文件夹 ID，根节点时为固定特殊值（如 0 或 null），需统一约定。  
  - `name`：文件夹名称，支持中文，长度建议 1~128。  
  - `path`：可选的冗余路径字段（如 `/root/部门A/项目X`），本期可选，若引入需在第 14 章中论证。  
  - `sortOrder`：排序权重，整型，越小越靠前。  
  - `ownerDeptId`：所属部门 ID，用于数据范围和责任主体。  
  - `ownerUserId`：创建者/负责人用户 ID。  
  - `inheritPermission`：是否继承上级文件夹权限（布尔）。  
  - `remark`：备注，长度建议 ≤ 512。  
  - `extensionData`（可选）：扩展 JSON 字段，对齐现有 `extensionData` 设计（如有），通过 `JacksonTypeHandler` 存储。
  - 审计与删除：`createdAt`、`createdBy`、`updatedAt`、`updatedBy`、`deleted`。

- **字段约束（设计）**
  - `name` 非空，去除首尾空格后不能为空。
  - 同一 `parentId` 下 `name` 必须唯一（同级唯一规则）。
  - `sortOrder` 可为空，默认值为创建时间顺序或 0。

- **与其他对象关系（设计）**
  - 与 `FolderPermission` / `FolderGrant` / `FolderManager` / `FolderFavorite` / `FolderOperationLog` 为一对多关系。
  - 与未来 `File` 对象为一对多关系（一个 folder 下有多个文件）。

- **生命周期（设计）**
  - 创建：新建时写入审计字段与层级信息。
  - 编辑：修改名称、备注、排序、继承标志等。
  - 移动：变更 `parentId`（以及可能的 `path`），影响子树。
  - 复制：创建新节点并复制子树结构与授权（根据规则）。
  - 删除：直接删除（物理删除或逻辑删除，需在第 10 章中明确策略）。

- **删除策略（设计）**
  - 本期按照“直接删除”原则，删除文件夹时：  
    - 必须删除该节点及其所有子孙节点。  
    - 对其关联的授权记录、管理员、收藏、操作日志做合理处理（授权/收藏可删除或标记失效，操作日志保留）。

- **审计字段要求（设计）**
  - 使用现有 `MyMetaObjectHandler` 自动填充审计字段。
  - 对批量操作（批量删除、批量移动）要求统一记录操作人和时间。

### 5.2 FolderPermission（文件夹权限点定义）
- **含义（设计）**
  - 定义文件夹级别的“操作种类”，例如查看、管理、授权、导出等，为权限矩阵中的列。

- **核心字段（设计）**
  - `code`：权限点编码（如 `FOLDER_VIEW`, `FOLDER_EDIT`, `FOLDER_DELETE`, `FOLDER_MOVE`, `FOLDER_COPY`, `FOLDER_GRANT`, `FOLDER_EXPORT`, `FOLDER_FAVORITE`）。  
  - `name`：权限点名称（中文）。  
  - `description`：说明。  
  - `builtin`：是否内置（避免被误删）。  

- **与其他对象关系（设计）**
  - 与 `FolderGrant` 为多对多关系的一方（另一方为“授权主体 + 文件夹”）。

- **生命周期与审计（设计）**
  - 本期可作为枚举或常量实现，无需单独建表；如未来希望通过配置管理，则可扩展为表结构，复用审计字段。

### 5.3 FolderGrant（文件夹授权关系）
- **含义（设计）**
  - 描述“对某个文件夹，某个授权主体（部门/角色/用户）拥有哪些权限点及其生效时间段”。

- **核心字段（设计）**
  - `id`：主键。  
  - `folderId`：目标文件夹 ID。  
  - `subjectType`：授权主体类型（枚举：`DEPARTMENT` / `ROLE` / `USER`）。  
  - `subjectId`：授权主体 ID（部门 ID、角色编码/ID、用户 ID）。  
  - `permissionCode`：权限点编码（参照 `FolderPermission.code`）。  
  - `inherit`：是否对下级文件夹生效（用于跨层级授权）。  
  - `effectiveFrom`：生效开始时间（可空，空代表立即生效）。  
  - `effectiveTo`：生效结束时间（可空，空代表长期有效）。  
  - `explicitDeny`：是否显式拒绝（增强项，可选；本期可先不落地，仅在文档中设计）。  
  - 审计与删除：同上。

- **字段约束（设计）**
  - `(folderId, subjectType, subjectId, permissionCode)` 组合唯一（避免重复授权）。
  - `effectiveFrom < effectiveTo`（若均不为空）。

- **与其他对象关系（设计）**
  - 关联 `Folder`（多对一）。
  - 关联授权主体实体（`SysDepartment`、`SysRole`、`SysUser`）——仅通过 ID 引用。

- **生命周期（设计）**
  - 创建：由拥有授权权限的用户发起（如 `FOLDER_ADMIN`/`SUPER_ADMIN`/folder 管理员）。
  - 修改：变更权限点、有效期等。
  - 失效：到期自动视为无效，或人工取消授权。
  - 删除：逻辑删除或物理删除均需保留审计痕迹。

### 5.4 FolderManager（文件夹管理员）
- **含义（设计）**
  - 描述某个用户在某个文件夹上的“管理者身份”，具备一定范围内的管理权限（如可为该文件夹及其子树配置授权、管理子文件夹）。

- **核心字段（设计）**
  - `id`：主键。  
  - `folderId`：文件夹 ID。  
  - `userId`：管理员用户 ID。  
  - `manageScope`：管理范围（如：仅当前节点 / 当前节点及子树）。  
  - `description`：说明。  
  - 审计与删除：同上。

- **与其他对象关系（设计）**
  - 关联 `Folder` 与 `SysUser`。
  - 在权限判定时，folder 管理员可作为一种“角色”参与简化判断。

### 5.5 FolderFavorite（文件夹收藏）
- **含义（设计）**
  - 记录用户对文件夹的收藏关系，方便在前端“我的收藏”中快速访问常用目录。

- **核心字段（设计）**
  - `id`：主键。  
  - `folderId`：文件夹 ID。  
  - `userId`：收藏人 ID。  
  - `favoriteTime`：收藏时间。  
  - `remark`：备注（可选）。  

- **字段约束（设计）**
  - `(folderId, userId)` 组合唯一（收藏去重规则）。  

### 5.6 FolderOperationLog（文件夹操作日志）
- **含义（设计）**
  - 存储与文件夹相关的关键操作记录，用于后续审计和追踪。

- **核心字段（设计）**
  - `id`：主键。  
  - `folderId`：受影响文件夹 ID。  
  - `operationType`：操作类型（如：CREATE、UPDATE、RENAME、MOVE、COPY、DELETE、GRANT、REVOKE、EXPORT、FAVORITE、UNFAVORITE）。  
  - `operatorUserId`：操作人用户 ID。  
  - `operatorDeptId`：操作人所属部门 ID。  
  - `operationTime`：操作时间。  
  - `requestId`：关联请求的 traceId/请求 ID（便于与系统日志关联）。  
  - `detailBefore`：操作前关键字段快照（JSON）。  
  - `detailAfter`：操作后关键字段快照（JSON）。  
  - `remark`：备注。

- **与其他对象关系（设计）**
  - 关联 `Folder` 与 `SysUser`/`SysDepartment`。
  - 可与 future log 模块的统一审计表打通。


## 6. 文件夹树设计

### 6.1 全局根节点约束（设计）
- 仅存在 **一个全局根节点**，表示系统文档的入口目录：
  - 根节点可采用固定 ID（如 `0` 或一个固定的雪花 ID），需在实现时统一常量。
  - 根节点不可被删除、移动或复制。

### 6.2 层级深度与无限层级（设计）
- **理论层级**：支持业务上的“无限层级”，以满足复杂项目/部门结构需要。
- **建议最大层级**：建议在实现中限制最大层级（如 8~10 层），超出时拒绝创建或移动：
  - 防止深度过大导致查询复杂度与用户体验下降。
  - 便于界面展示与导出。

### 6.3 父子关系存储方案（设计）
- **现状参考**：现有 `SysDepartment` 已通过父子字段实现树形结构（从实体可见层级与部门关系字段）。
- **设计方案**：folder 模块采用 **邻接表模型（Adjacency List）**：
  - 使用 `parentId` 字段表征父子关系。
  - 依赖递归查询/多次查询拼接树结构。
- **原因**：
  - 与现有项目实体风格一致，便于团队理解与维护。
  - 不引入 Postgres 专用特性作为强约束（如 `ltree`），降低迁移成本。
  - 复杂查询（如深层搜索）可通过可选增强项优化（见第 14 章）。

### 6.4 排序规则（设计）
- 同一父节点下，通过 `sortOrder` 字段进行排序：
  - 默认按 `sortOrder` 升序，若相等则按 `createdAt` 升序。
  - 在移动或复制操作后，可重新编排 `sortOrder` 以避免过大间隙。

### 6.5 同级唯一规则（设计）
- 在数据库层面建议建立唯一索引：
  - `UNIQUE(parent_id, name, deleted=false)` 或等价逻辑。
- 在业务层面，创建/重命名/移动时均需校验新父节点下是否已有同名文件夹。

### 6.6 路径展示规则（设计）
- 展示格式建议为：`根目录 / 一级 / 二级 / ... / 当前节点`。
- 实现方式建议：
  - **最小方案**：按需查询父链（自底向上多次查询）并在内存中拼装路径。
  - **增强方案**（可选）：引入冗余 `path` 字段存储路径字符串，变更时更新整棵子树的 `path`；在第 14 章中详细论证。

### 6.7 防循环引用规则（设计）
- 在移动/复制时，必须校验：
  - 目标父节点不是当前节点本身。
  - 目标父节点不在当前节点的子树中（通过遍历/查询子节点列表校验）。
- 若违反上述规则，返回业务异常，错误码建议使用 `ErrorCode.PARAM_INVALID` 或新增 folder 专用错误码。

### 6.8 子树移动规则（设计）
- **主流程**：
  - 校验当前用户对源文件夹具备移动权限，对目标父文件夹具备创建权限。
  - 校验防循环引用与同级唯一（新父下无同名）。
  - 更新被移动节点及其子孙节点的 `parentId`（仅根节点变更时，其子节点 `parentId` 保持指向原结构）。
  - 若引入 `path` 或层级缓存，则同时更新相关字段。
- **权限与继承影响**：
  - 若源文件夹启用了继承（`inheritPermission=true`），移动到新父节点后，权限继承链需按新路径重新计算。
  - 对该文件夹及子树上的显式授权保持不变，仅继承部分发生变化。

### 6.9 子树复制规则（设计）
- **主流程**：
  - 复制指定文件夹及其所有子孙文件夹结构，生成新的 ID 与 `parentId`。
  - `name` 冲突时可自动追加后缀（如“(复制)”）或提示用户修改。
- **权限与继承影响**：
  - 建议：复制结构时，只复制结构与基础元数据，不复制显式授权，避免大量错误继承；  
    可选增强：支持“复制时一并复制授权”，但需明确风险。

### 6.10 子树删除规则（设计）
- 删除文件夹时：
  - 必须删除其自身及所有子孙文件夹（通过递归/批量操作实现）。
  - 对应的授权记录、管理员记录、收藏记录：
    - 可采用逻辑删除或物理删除，但推荐保留操作日志。
  - 若后续 file 模块已存在，则需处理该子树下所有文件的删除策略（本期仅在设计层标记，实际实现由 file 模块配合）。


## 7. 权限模型设计

> 本节在现有角色模型与 Sa-Token 集成方式基础上，设计 folder 专用权限点、矩阵、继承/覆盖与缓存策略，保持“角色级 + 资源级”组合的模式。

### 7.1 文件夹级权限点（设计）
- 建议的权限点清单（对应 `FolderPermission.code`）：
  - `FOLDER_VIEW`：查看文件夹及其基础信息、文件清单。
  - `FOLDER_EDIT`：编辑文件夹元数据（名称、备注、排序等）。
  - `FOLDER_CREATE`：在指定父文件夹下创建子文件夹。
  - `FOLDER_DELETE`：删除文件夹及其子树。
  - `FOLDER_MOVE`：移动文件夹到新父目录。
  - `FOLDER_COPY`：复制文件夹子树。
  - `FOLDER_GRANT`：对文件夹进行授权（新增/修改/删除 `FolderGrant`）。
  - `FOLDER_MANAGER_CONFIG`：设置/取消 `FolderManager`。
  - `FOLDER_EXPORT`：导出文件清单。
  - `FOLDER_SEARCH`：按条件检索/筛选文件夹或其文件。
  - `FOLDER_LOG_VIEW`：查看文件夹操作日志。
  - `FOLDER_FAVORITE`：收藏/取消收藏文件夹。

### 7.2 管理员权限（设计）
- `SUPER_ADMIN`：
  - 对所有文件夹拥有所有权限点，无需额外授权。
- `FOLDER_ADMIN`：
  - 对所有文件夹拥有大部分管理权限（除极少数平台级操作，如根节点保护）。
  - 可进行授权与管理员配置。
- `DEPT_MANAGER`：
  - 对本部门及其子部门负责的文件夹具备较高权限（如创建、编辑、导出等），具体权限矩阵见下。
- folder 管理员（FolderManager）：
  - 对被管理文件夹及其范围内的资源，具备一定的授权与管理能力，具体以配置的 `manageScope` 与映射到权限点的规则为准。

### 7.3 授权对象（设计）
- 授权对象三类：
  - 部门（`DEPARTMENT`）：对应 `SysDepartment`。
  - 角色（`ROLE`）：对应 `SysRole` 或 `RoleCodeEnum`。
  - 用户（`USER`）：对应 `SysUser`。
- FolderGrant 中通过 `(subjectType, subjectId)` 抽象表示授权主体。

### 7.4 权限继承规则（设计）
- 默认继承：
  - 若 `Folder.inheritPermission=true`，则该文件夹默认继承父文件夹的权限集合。
- 继承链：
  - 从根节点向下传递，遇到显式授权时进行合并或覆盖。
- 关闭继承：
  - 若某节点设置 `inheritPermission=false`，则仅根据自身与其子树中的显式授权判定，不自动合并父级权限（但 `SUPER_ADMIN` / `FOLDER_ADMIN` 的全局特权不受影响）。

### 7.5 权限覆盖与冲突规则（设计）
- 建议的判定优先级（从高到低）：
  1. 显式拒绝（`explicitDeny=true` 的 FolderGrant，若本期不落地则视为增强项）。  
  2. 当前节点的显式允许授权。  
  3. 继承自上级节点的允许授权。  
  4. 角色/部门默认权限策略（如 `FOLDER_ADMIN` 默认全局通过）。  
- 当同一用户通过多重路径获得不同权限集时，按“允许取并集，拒绝覆盖允许”的原则执行（若启用显式拒绝）。

### 7.6 权限有效期规则（设计）
- 若 `effectiveFrom` 或 `effectiveTo` 不为空：
  - 当前时间 < `effectiveFrom` → 授权尚未生效。  
  - 当前时间 > `effectiveTo` → 授权已过期。  
- 权限判定时需过滤掉未生效/已过期的授权记录。

### 7.7 权限冲突规则与显式拒绝（设计）
- 本期基础方案可**不实现显式拒绝**，仅通过“允许权限”的合并解决问题；  
  显式拒绝作为增强项：
  - 典型场景：某用户归属于一个有查看权限的角色，但需要对某个敏感目录进行单独屏蔽。
  - 实现侵入性较大，需要在 FolderGrant 与权限判定逻辑中显式处理，建议在第 14 章中作为增强建议，不强制当前实现。

### 7.8 权限校验时机（设计）
- Controller 层：
  - 使用现有 `TokenInterceptor` 保证所有请求均已登录并具备 `UserContext`。
  - 可通过注解（如未来扩展 `@FolderPermission("FOLDER_EDIT")`）结合 AOP 校验，或在 Service 层统一显式调用权限检查方法。
- Service 层：
  - 在执行任何会修改数据的操作前进行权限判定。
  - 在读取敏感数据时（如导出、日志查询）同样需要权限校验。

### 7.9 Redis 缓存策略（设计）
- 适合缓存的内容：
  - 用户在某个文件夹上的权限集合（含继承链计算结果）。
  - 某文件夹的聚合权限信息（如可访问的用户/角色列表，用于权限管理界面）。
- Key 命名建议（示例）：
  - `folder:perm:user:{userId}:{folderId}`：某用户对某文件夹的权限集合。  
  - `folder:perm:folder:{folderId}`：某文件夹聚合后的授权信息缓存。  
- 失效策略：
  - 授权变更（FolderGrant 变更）时，失效与该 folder 相关的缓存。
  - 角色/部门调整时（由 auth 模块触发或通过事件总线/手动接口），失效相关用户的权限缓存。
  - 移动/删除子树时，对整棵子树相关缓存进行批量失效。
- 一致性风险与处理：
  - 短时间内缓存与数据库授权不一致可能导致“短暂脏读”，建议在敏感操作中强制绕过缓存或重新加载。

### 7.10 Sa-Token 接入点与与现有权限控制衔接（设计）
- 登录与用户信息：
  - 保持现有 `StpUtil` + `UserContext` 使用方式，folder 模块仅在此之上扩展权限逻辑。
- 按钮权限与角色控制：
  - 在 Controller 层可继续使用 `@SaCheckRole("FOLDER_ADMIN")` 控制高危接口（如授权与删除）。
  - 细粒度的操作级权限通过 FolderGrant 与 Redis 缓存实现，由 Service 层统一判定。


## 8. 功能需求清单

> 本节为 folder 模块提供的核心功能点需求说明，每个功能采用统一结构描述。为节省篇幅，下文仅在首次功能中展开说明模板，后续以要点方式给出。

### 8.1 新建文件夹
- **功能目标**：在指定父文件夹下创建新的子文件夹。
- **参与角色**：登录用户（权限取决于角色/授权），`SUPER_ADMIN`、`FOLDER_ADMIN`、相关 folder 管理员、具备 `FOLDER_CREATE` 的用户。
- **前置条件**：
  - 用户已登录；`UserContext` 有效。
  - 目标父文件夹存在且未被删除。
  - 用户对目标父文件夹具有 `FOLDER_CREATE` 权限。
- **后置结果**：
  - 新增一条 `Folder` 记录。
  - 审计字段正确填充。
  - 记录一条 `FolderOperationLog`（类型为 CREATE）。
- **主流程（设计）**：
  1. 接收请求（父文件夹 ID、名称、备注等）。
  2. 校验参数合法性和层级深度。
  3. 校验权限（含继承权限与显式授权）。
  4. 校验同级唯一（父节点下无同名）。
  5. 写入 `Folder` 表，填充审计字段与排序字段。
  6. 写入操作日志。
- **异常流程**：
  - 父文件夹不存在或已删除 → 返回资源不存在错误。
  - 权限不足 → 返回 `ErrorCode.PERMISSION_DENIED`。
  - 同名冲突 → 返回业务错误（可使用通用参数冲突错误码或新增 folder 专用错误码）。
- **权限要求**：
  - 至少具备 `FOLDER_CREATE`，且所在角色/部门/用户组合在该父目录上的授权允许创建。
- **日志要求**：
  - 系统日志中记录请求与结果；业务操作日志中记录创建详情（包含 folder 关键信息与操作人）。
- **数据影响范围**：
  - 写入一条 `Folder`，影响相关缓存（若缓存父目录的子列表）。

### 8.2 编辑文件夹
- **功能目标**：修改文件夹的备注、继承标志、排序等非名称字段。
- **参与角色**：具备 `FOLDER_EDIT` 权限的用户。
- **前置条件**：文件夹存在、未被删除；用户具备相应权限。
- **后置结果**：更新 `Folder` 信息，写入操作日志（UPDATE）。
- **异常流程**：权限不足、目标不存在、更新冲突（版本控制可作为增强）。

### 8.3 重命名
- **功能目标**：修改文件夹名称。
- **参与角色**：具备 `FOLDER_EDIT` 权限的用户。
- **关键规则**：
  - 同级唯一校验。
  - 禁止仅大小写不同但逻辑上视为相同名称的情况（视具体字符集与业务要求决定）。
- **数据影响**：更新 `Folder.name`（可能同时更新 `path` 冗余字段），记录操作日志。

### 8.4 删除
- **功能目标**：删除指定文件夹及其子树。
- **参与角色**：具备 `FOLDER_DELETE` 权限的用户，通常为管理员角色。
- **关键规则**：
  - 删除前需校验是否允许删除根节点（通常禁止）。
  - 子树整体删除，避免悬挂节点。
- **数据影响**：
  - 删除 `Folder` 子树记录及关联授权、管理员、收藏记录（策略见第 10 章）。
  - 记录操作日志（DELETE），日志本身不删除。

### 8.5 批量删除
- **功能目标**：一次删除多个文件夹（可能包含重叠子树）。
- **关键规则**：
  - 合并子树范围，避免重复处理相同节点。
  - 需要幂等设计（重复提交不会导致异常）。
- **数据影响**：
  - 可能涉及大量记录，需考虑性能与事务边界。

### 8.6 移动
- **功能目标**：将一个文件夹移动到另一个父文件夹下。
- **关键规则**：
  - 防循环引用。
  - 同级唯一。
  - 权限校验：对源目录和目标父目录均具备相应权限（`FOLDER_MOVE` + 对目标的 `FOLDER_CREATE`）。

### 8.7 复制
- **功能目标**：复制指定文件夹及其子树结构。
- **关键规则**：
  - 同名冲突处理策略（自动加后缀或提示用户）。
  - 授权复制策略（建议默认不复制授权，仅复制结构；增强项可选“复制授权”）。

### 8.8 查看文件夹树
- **功能目标**：按层级展示文件夹树结构（全量或按部门/权限过滤）。
- **关键规则**：
  - 仅返回用户有查看权限的节点（可按“可访问子树”裁剪树）。
  - 支持按层级懒加载（只加载部分层级，需前后端协商）。

### 8.9 查看文件夹详情
- **功能目标**：展示单个文件夹的详细信息与关联信息（如授权、管理员、收藏状态）。
- **关键规则**：
  - 校验 `FOLDER_VIEW` 权限。
  - 可选包含当前用户对该文件夹的权限集合信息，方便前端控制按钮显示。

### 8.10 文件夹授权
- **功能目标**：为文件夹配置部门/角色/用户授权。
- **参与角色**：`SUPER_ADMIN`、`FOLDER_ADMIN`、具备 `FOLDER_GRANT` 的 folder 管理员等。
- **关键规则**：
  - 接口支持批量授权与批量撤销。
  - 支持设置/修改生效时间。
  - 授权变更后，需要刷新或失效相关权限缓存。

### 8.11 文件夹管理员设置
- **功能目标**：设置/撤销某用户为指定文件夹的管理员。
- **关键规则**：
  - 管理范围配置（仅当前节点 / 当前及子树）。
  - 与 FolderGrant 的关系：管理员缺省具备哪些权限点需明确定义。

### 8.12 子文件夹权限继承控制
- **功能目标**：在某个节点上配置是否继承上级权限。
- **关键规则**：
  - 设置为“不继承”时，需要清晰指引管理员确保显式授权足够。
  - 切换继承标志可能影响整棵子树的有效权限，需要重新计算或清缓存。

### 8.13 导出文件清单
- **功能目标**：按文件夹导出文件清单（如 Excel）。
- **关键规则**：
  - 仅允许具备 `FOLDER_EXPORT` 权限的用户导出。
  - 导出数据应受数据范围与文件夹授权双重限制。

### 8.14 搜索 / 条件筛选
- **功能目标**：按名称、路径、部门、创建时间等条件筛选文件夹，或以 folder 为入口筛选文件。
- **关键规则**：
  - 搜索结果仅包含用户有查看权限的文件夹。
  - 可提供分页结果（结合第 10/12 章的分页约定）。

### 8.15 操作日志查询
- **功能目标**：查询与指定文件夹相关的操作日志。
- **关键规则**：
  - 仅允许具备 `FOLDER_LOG_VIEW` 或更高权限（如管理员）的用户查看。
  - 支持按操作类型、时间范围、操作人筛选。

### 8.16 收藏文件夹
- **功能目标**：用户可以收藏常用的文件夹，并在“我的收藏”中快速访问。
- **关键规则**：
  - 收藏前需校验用户对该文件夹至少具备 `FOLDER_VIEW` 权限。
  - `(folderId, userId)` 唯一，重复收藏保持幂等。
  - 取消收藏后，记录操作日志（UNFAVORITE）。


## 9. 规则设计

### 9.1 命名规则（设计）
- 文件夹名称：
  - 允许中英文、数字、常见符号（如空格、下划线、连字符），不建议包含控制字符与不可见字符。
  - 禁止纯空格或长度超过 128。

### 9.2 同级唯一规则（设计）
- 数据库层唯一索引 + 业务校验双重保证。
- 对于逻辑删除的记录，可视需求决定是否参与唯一性（推荐：只对 `deleted=false` 生效）。

### 9.3 编码规则（设计）
- 如需为文件夹引入业务编码（如 `FOL-20260001`），建议作为增强字段，在本期设计中预留，但不强制实现。

### 9.4 删除规则（设计）
- 单个/批量删除均采用“直接删除”策略（逻辑或物理在第 10 章中确定），**不进入回收站**。
- 删除操作不可逆，需通过权限和确认流程控制风险。

### 9.5 移动/复制规则（设计）
- 遵守防循环与同级唯一。
- 对授权与收藏的处理需清晰（移动保留、复制默认不复制授权，收藏不随复制自动扩散）。

### 9.6 导出规则（设计）
- 导出文件清单时：
  - 受数据范围与文件夹授权双重限制。
  - 输出字段需与后续接口文档一致（如文件名、路径、所属部门、创建人、时间等）。

### 9.7 搜索匹配规则（设计）
- 支持按名称前缀/模糊匹配。
- 支持按路径包含、部门、创建人、创建时间等条件组合。

### 9.8 收藏去重规则（设计）
- 通过唯一键 `(folderId, userId)` 保证单个用户不会对同一文件夹同时存在多条收藏记录。

### 9.9 日志记录规则（设计）
- 所有涉及 CRUD 与授权的操作应记录 `FolderOperationLog`，且带有 traceId 以对齐系统日志。

### 9.10 权限继承与生效/失效规则（设计）
- 权限按继承链从根到当前节点叠加，显式授权优先。
- 有效期与显式拒绝（如启用）对继承链有最高优先权。

### 10.2 新增表（设计）

  - `created_at`、`created_by`、`updated_at`、`updated_by`、`deleted` 字段与现有实体一致。

### 10.3 扩展现有表（设计/待确认）
- 如未来希望将 folder 操作日志统一落在通用 log 表中，可考虑：
  - 在通用 log 表中增加 `object_type`、`object_id` 字段用于多模块共用。
  - 该方案目前在代码中尚未见到实现，需在后续设计 log 模块时统一规划。

### 10.4 索引与唯一索引建议（设计）
- `folder` 表：
  - 主键索引：`id`。  
  - 唯一索引：`(parent_id, name, deleted)`。  
  - 普通索引：`parent_id`、`owner_dept_id`、`owner_user_id`、`created_at`。
- `folder_grant` 表：
  - 唯一索引：`(folder_id, subject_type, subject_id, permission_code, deleted)`。
  - 非唯一索引：`subject_type, subject_id`（按授权主体逆向查询）。
- `folder_favorite` 表：
  - 唯一索引：`(folder_id, user_id, deleted)`。
  - 非唯一索引：`user_id`（查询“我的收藏”）。

### 10.5 审计与逻辑删除字段（设计）
- 所有业务表均应包含：
  - `created_at`、`created_by`、`updated_at`、`updated_by`、`deleted`。
  - 与 `MyMetaObjectHandler` 及 MyBatis-Plus 逻辑删除配置对齐。

### 10.6 JSONB 使用建议（设计）
- 仅在以下场景推荐考虑 JSONB：
  - `FolderOperationLog` 中的 `detailBefore`、`detailAfter` 字段。
  - `Folder` 的扩展字段 `extensionData`。
- 收益：
  - 提高灵活性，支持未来增加更多字段而无需改表。
- 复杂度：
  - 查询与索引设计更复杂，调试成本上升。
- 建议：
  - 本期可在日志表中采用 JSONB 存储详情；  
  - `Folder` 的扩展属性如无明确需求，可暂不引入 JSONB，将其作为增强项。

### 10.7 仍需从 DDL 确认的事项（待确认）
- 现有 `sys_*` 表的真实列名、索引与约束（与实体推断可能存在差异）。
- 数据库中是否已有与 folder 功能相关的历史表或预留表名。


## 11. 缓存与性能设计

### 11.1 适合缓存的内容（设计）
- 文件夹树结构（或其局部）：
  - 如常用目录树、指定部门/项目根下的子树。
- 用户在某文件夹或子树上的权限集合：
  - 降低频繁的权限计算开销。

### 11.2 不适合缓存的内容（设计）
- 高频更新、强一致要求的数据：
  - 刚刚授权/撤销授权的目录的即时状态，如对安全要求极高的场景，可避免过度缓存。
- 大体积的日志或历史操作明细：
  - 建议仍然直接从数据库分页查询。

### 11.3 文件夹树缓存策略（设计）
- 可选方案：
  - 缓存全量树：适合节点数量有限（几千级）的场景，但更新时需要全量重建或大批量失效。
  - 缓存局部树：按根节点/部门/项目分片缓存，减少更新范围。
- 建议最小方案：
  - 先以数据库实时查询为主，针对热点子树进行短期缓存；  
  - 根据实际数据量与访问模式再评估是否引入全量树缓存。

### 11.4 权限缓存策略（设计）
- 缓存 key 与失效策略参见第 7.9 节。
- 在 1200 峰值并发下：
  - 对频繁访问的用户/目录组合，缓存权限结果可显著降低数据库与计算压力。
  - 授权变更相对较少，适合采用“写时失效 + 读时重建”的策略。

### 11.5 一致性风险（设计）
- 缓存延时导致的短暂权限不一致：
  - 授权刚撤销但缓存未失效 → 短时间内用户仍能访问。
- 缓解措施：
  - 对于高风险操作（删除、导出、授权变更），可以在操作前强制绕过缓存或即时刷新缓存。

### 11.6 深层树性能风险（设计）
- 深层树查询可能导致递归或多次数据库往返：
  - 需在实现时限制层级深度，并对深层树上的批量操作做性能测试。

### 11.7 批量授权性能风险（设计）
- 在一次操作中为大量文件夹/主体授予/撤销权限，可能带来写放大与缓存失效风暴：
  - 建议在设计接口时限制单次操作的对象数量。  
  - 对大批量操作采用异步或后台任务（见增强项）。


## 12. 接口设计前置说明

### 12.1 接口域划分建议（设计）
- 将 folder 模块的接口划分为：
  - `FolderController`：文件夹基础 CRUD 与树查询。
  - `FolderAuthController`：授权与管理员配置。
  - `FolderFavoriteController`：收藏相关接口。
  - `FolderLogController`：操作日志查询。

### 12.2 Controller 粒度建议（设计）
- 遵循现有 `auth` Controller 风格：
  - RESTful 风格，统一使用 `ApiResponse` 包裹。
  - 使用 `/api/v1/folders/...` 作为路径前缀。

### 12.3 DTO / VO 设计建议（设计）
- DTO：
  - 请求 DTO 使用小驼峰 JSON 字段，对应 Java 字段。
  - 引入专用 DTO，如 `FolderCreateRequest`, `FolderUpdateRequest`, `FolderMoveRequest` 等。
- VO：
  - 响应 VO 用于展示视图数据，如 `FolderDetailVO`, `FolderTreeNodeVO`, `FolderPermissionVO`。

### 12.4 分页查询约定（设计）
- 建议统一使用：
  - 请求：`page`（页码，从 1 开始）、`size`（每页大小）。  
  - 返回：复用 `PageResponse<T>`，包含 `list` 与 `pagination` 信息。

### 12.5 树查询返回结构建议（设计）
- 节点结构：
  - `id`, `name`, `children`, `hasChildren`, `path`, `depth`, `sortOrder` 等。
- 支持：
  - 全量树与懒加载模式（通过是否携带 `children` 决定）。

### 12.6 批量操作入参建议（设计）
- 对批量删除/授权/收藏等：
  - 使用 ID 列表作为参数，如 `folderIds: List<Long>`、`userIds: List<Long>`。
  - 为大批量操作预留异步能力（如传入任务 ID，异步执行）。

### 12.7 错误码域建议（设计）
- 在现有 `ErrorCode` 基础上：
  - 预留一段 folder 专用错误码区间，用于表达同级重名、防循环引用、层级过深等特有错误。

### 12.8 幂等建议（设计）
- 对以下操作建议支持幂等：
  - 收藏/取消收藏（重复调用不报错）。  
  - 批量授权/撤销（对已存在/不存在的记录做幂等处理）。  
  - 导出任务创建（可通过幂等键或请求参数 hash 防止重复任务）。

### 12.9 审计字段返回建议（设计）
- 在用于管理/审核的接口中，可以选择性返回：
  - `createdAt`、`createdBy`、`updatedAt`、`updatedBy`，便于管理员查看责任人和变更历史。


## 13. MVC 落地建议

### 13.1 包结构建议（设计）
- 建议在 `com.example.biddoc.folder` 下建立：
  - `controller`、`service`、`service.impl`、`mapper`、`entity`、`dto`、`convertor`、`constant` 等子包，与 `auth` 保持一致。

### 13.2 Controller 职责边界（设计）
- 仅负责：
  - 参数接收与校验（JSR 303 + 自定义校验）。
  - 调用 Service 并统一封装 `ApiResponse`。
  - 不直接访问 Mapper 或操作事务。

### 13.3 Service 职责边界（设计）
- 负责：
  - 业务逻辑编排、权限校验、数据范围控制。
  - 调用多个 Mapper 完成复杂业务。
  - 控制事务边界（使用 `@Transactional`）。

### 13.4 Mapper 职责边界（设计）
- 仅负责单表/少量多表的数据访问，不包含业务判断。
- 统一使用 MyBatis-Plus `BaseMapper` 与 LambdaWrapper 风格。

### 13.5 Entity / DTO / VO / Convert 边界（设计）
- Entity：与数据库表一一对应，不暴露给外部。
- DTO：接收外部请求参数。
- VO：对外响应视图对象。
- Convertor：集中管理 Entity/DTO/VO 之间的转换，保持 `auth` 模块风格。

### 13.6 Enum / Constants / ExceptionCode 设计建议（设计）
- Enum：
  - 定义 `FolderPermissionCodeEnum`, `FolderSubjectTypeEnum`, `FolderOperationTypeEnum` 等。
- Constants：
  - 定义根节点 ID、路径分隔符等常量。
- ExceptionCode：
  - 在 `ErrorCode` 或等效类中加入 folder 专用错误码。

### 13.7 事务边界建议（设计）
- 单个 Service 方法内完成一个完整的业务操作：
  - 如移动/删除/复制子树，需考虑事务超时与锁粒度。
- 对大批量操作可拆分为批次或采用异步任务。

### 13.8 slf4j 日志打印建议（设计）
- 在关键路径（移动/复制/删除/授权）中打印 info 或 debug 日志：
  - 日志中包含 folderId、操作人、结果与耗时。
  - 避免打印过大对象（完整树结构等）。

### 13.9 Docker 部署注意事项（设计）
- 现状：仓库中尚未提供 Dockerfile 或部署脚本。
- 建议：
  - 在未来 Docker 化时，确保：
    - 日志输出到 stdout，以便容器平台采集。
    - 配置通过环境变量覆盖（包括数据库与 Redis 连接）。
    - 与 Redis、PostgreSQL 的网络与认证配置安全可控。


## 14. 可选增强项（谨慎）

### 14.1 PostgreSQL ltree / 路径冗余字段（设计）
- **引入原因**：提升深层树结构的查询与搜索性能。
- **解决的问题**：
  - 快速查询某节点的全量子树或祖先路径。
- **侵入性**：
  - 使用 `ltree` 类型需要在 DDL 与 ORM 映射层增加支持，对迁移和兼容性有一定影响。
- **替代方案**：
  - 采用字符串 `path` 字段 + 索引 + 前缀匹配。
- **当前是否建议立即落地**：
  - 视预期数据量与查询复杂度决定。对于中小型项目，可先使用邻接表 + 简单 path 字段，`ltree` 可作为后期优化手段。

### 14.2 异步日志写入（设计）
- **引入原因**：减轻主业务线程写操作日志表的延迟。
- **解决的问题**：
  - 高并发下操作日志写入导致的响应时间抖动。
- **侵入性**：
  - 需要引入消息队列或异步任务执行机制。
- **替代方案**：
  - 采用批量插入或简单线程池异步提交。
- **当前建议**：
  - 在当前规模与场景下，可先使用同步写入；当操作日志量增大时再评估。

### 14.3 异步导出任务（设计）
- **引入原因**：导出大量文件清单可能耗时较长。
- **解决的问题**：
  - 避免长时间占用 HTTP 连接与线程。
- **侵入性**：
  - 需要任务调度/任务表等配套设施。
- **替代方案**：
  - 限制单次导出数据量，在同步接口内完成。
- **当前建议**：
  - 视实际导出规模再决定，本期可先实现同步导出。

### 14.4 权限缓存预热（设计）
- **引入原因**：减少首次访问时权限计算开销。
- **解决的问题**：
  - 新用户首次访问大量目录时的冷启动性能。
- **侵入性**：
  - 需要在登录或角色变更时预计算权限。
- **替代方案**：
  - 懒加载（首次访问时再计算并写入缓存）。
- **当前建议**：
  - 可以作为优化选项，但非必需。


## 15. 非功能需求

### 15.1 性能（设计）
- 在目标规模（约 3000 用户、1200 峰值并发）下：
  - 常规 CRUD 接口 P95 响应时间建议 < 200ms。
  - 树查询与导出操作在数据量可控范围内保持 < 1s。

### 15.2 安全（设计）
- 所有修改操作必须经过身份认证与权限校验。
- 对关键操作保留操作日志与 traceId。

### 15.3 审计（设计）
- 关键字段变化需在 `FolderOperationLog` 中完整记录（前/后值）。
- 日志保留周期与归档策略需在运维层进一步约定。

### 15.4 可维护性与可扩展性（设计）
- 代码结构按模块/分层清晰划分，避免跨层调用。
- 为未来 file、tag 等模块预留必要的扩展点（如 folderId 外键、路径字段）。

### 15.5 一致性与可用性（设计）
- 在缓存与数据库不一致的场景下，以数据库为最终权威。
- 关键写操作需采用事务保证内部一致性。

### 15.6 容灾恢复与多端适配影响（设计）
- 容灾恢复：
  - 依赖数据库备份与日志审计，folder 模块不额外引入专有容灾机制。
- 多端适配：
  - 接口设计保持端无关性（Web/移动端共享），通过统一的 JSON 格式暴露。


## 16. 风险、约束、禁止事项

### 16.1 技术约束与架构风险（现状/设计）
- 单体架构下，folder 模块与 auth/common 在同一进程内，模块之间需保持良好边界，避免循环依赖。
- 未来若要支持多租户、多项目，folder 树的单根设计可能需要扩展，需要提前在数据模型上预留租户/项目维度。

### 16.2 数据迁移与权限误配风险（设计）
- 在已有数据的前提下（如未来已有部门/项目结构），引入 folder 树可能需要对已有文件进行迁移与归档。
- 权限误配（如错误授权整个子树）可能导致严重的越权访问风险，需要：
  - 清晰的权限矩阵。
  - 界面上的确认与提示。
  - 操作日志可追溯。

### 16.3 树层级过深与批量操作风险（设计）
- 层级过深可能导致性能问题和用户体验问题，需要通过最大层级限制与 UI 设计控制。
- 批量删除/授权/移动等操作有放大效应，对性能和数据安全需特别关注。

### 16.4 缓存一致性风险（设计）
- 权限和树结构缓存若未及时失效，会导致短期不一致：
  - 需在实现时明确失效策略和关键路径绕过机制。

### 16.5 禁止事项（必须遵守）
1. **禁止虚构现有表、现有接口、现有鉴权逻辑**：  
   - 本文已将所有现状与设计清晰区分，后续实现需以真实代码与 DDL 为准。
2. **禁止把建议方案写成现状事实**：  
   - 所有设计与增强项不得在文档或实现中伪装为已有行为。
3. **禁止脱离 auth / file / log / search 等模块孤立设计 folder**：  
   - folder 必须与这些模块的现有/未来职责保持协同。
4. **禁止忽略 Sa-Token 与数据范围控制的既有实现**：  
   - 尽管数据范围尚未落地，但相关上下文已存在，folder 设计需考虑未来接入该能力。
5. **禁止忽略日志审计、缓存一致性、批量操作风险**：  
   - 所有关键操作都需要有日志与缓存策略。
6. **禁止无充分论证就引入重型中间件**：  
   - 如消息队列、分布式锁中间件等应在有明确收益且与整体架构协调时再引入。
7. **禁止把已排除功能重新纳入本期范围**：  
   - 回收站、多文件夹引用、审批流程、批量导入等明确不在本期范围内的功能不得在实现中暗含。
